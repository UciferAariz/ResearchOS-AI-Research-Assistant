The problem

Researchers can't keep up — and keyword search doesn't help

  • Volume: thousands of papers a week; nobody can read them all.
  • Discovery gap: keyword search misses conceptually-related work that uses different terminology.
  • Reading cost: deciding whether a paper is relevant still means opening and skimming the full PDF.
  • No grounding: generic chatbots hallucinate about papers instead of citing them.

Architecture

How it fits together

Next.js frontend (Vercel) FastAPI backend (AMD ROCm notebook) ────────────────────────── ────────────────────────────────────── search / chat / compare ──HTTPS──▶ ┌───────────────────────────────────┐ discover / upload / recs Cloudflare │ sentence-transformers │ Tunnel │ all-MiniLM-L6-v2 ──▶ AMD GPU │ │ (ROCm/HIP) │ ├───────────────────────────────────┤ │ ChromaDB (persistent vectors) │ │ arXiv / PubMed clients │ │ Fireworks AI (LLM: chat/compare) │ └───────────────────────────────────┘

GPU-bound embedding vs. LLM inference are cleanly separated; services wired via FastAPI dependency injection. ChromaDB persists vectors on disk across restarts.

Inside the backend, sentence-transformers runs the embeddings on the AMD GPU, ChromaDB persists the vectors on disk, and I pull papers from arXiv and PubMed. LLM inference for chat and comparison is served separately by Fireworks — so the latency-critical embedding path stays on AMD." (~18s)
